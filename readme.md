# **📄 PRD**

### 개요

1. 배경 및 문제 정의
    - B2B 영업 환경에서는 고객사로부터 수신하는 이메일 한 건에 ERP 수정 요청과 정책 질의 등이 복합적으로 혼재되어 있음
    - 현재 영업 담당자는 하나의 문의를 처리하기 위해 ERP 화면, 사내 매뉴얼, 타 부서 연락망 등 서로 다른 채널을 오가며 수작업으로 정보를 취합해야 하며, 이 과정에서 다음과 같은 문제 발생
        - **처리 병목 및 리드타임 지연**: 복합 문의 1건 처리에 평균적으로 n시간~n일이 소요되며, 출고 마감일을 넘기거나 발주 마감일을 넘겨 후속 물류 프로세스 전체가 연쇄적으로 지연되는 상황 발생 가능
        - **휴먼 에러로 인한 재무적 손실**: 수동 데이터 입력 과정에서 데이터 오기입, 출하 완료 건에 대한 주문 데이터 수정 등의 실수가 발생해 물류비 이중 청구 등 치명적인 손실 발생
        - **담당자 인지 과부하**: 비숙련 담당자의 경우 여러 시스템에서 정보를 교차 검증하고 수합해야 하므로 작업 과정에서 막대한 인지적 노력이 소모되어 고부가가치 업무에 집중하기 어려움
2. 개선 방향 및 목적
    - 고객의 자연어 이메일을 입력받아, ERP 트랜잭션 검증/실행(Worker A)과 매뉴얼 기반 지식 검색(Worker B)을 환각(Hallucination) 없이 병렬 처리하는 지능형 오케스트레이션 에이전트 구축
    - 단순 챗봇이나 RPA를 넘어, 사람의 최종 승인(Human-in-the-Loop)을 안전망으로 갖추면서도 반복적인 수작업을 자동화하는 하이브리드 AI 시스템 지향

### 목표

1. **비즈니스 목표**: B2B 영업 담당자, SCM/물류 관리자, 고객 지원팀 등
    1. 빠르고 정확한 응대를 통한 고객과의 신뢰도 구축
    2. 휴먼 에러로 인한 손해비용 절감
    3. 담당자 생산성 향상
2. **사용자 목표**: 발주 수량 및 납기 변경 요청, 규정 질의 등을 영업팀에 전달하는 고객사 담당자
    1. 문의에 대한 빠르고 정확한 답변 수령
    2. 문의 접수부터 처리까지의 시간 단축으로 후속 작업 적시 착수 가능

### 사용자 시나리오

1. SDS 영업 담당자 시나리오
    - 페르소나: SDS 영업 담당자로서, 10개 이상의 담당 고객사 관리, 일 평균 5건 이상의 고객 문의 이메일 처리
    - 개선 전 시나리오
        - 고객사의 복합 문의 이메일 수령
        - 요청 사항 항목별로 수동 분류 및 정리
        - 각 항목에 대해 처리
            - ERP 수정 요청: 해당 주문 건을 ERP에서 직접 검색 후, 수정 가능 여부(가용 재고, 납품 상태 등에 따라 결정)를 수동으로 확인하고 필드 수정
            - 정책 질의: 관련 매뉴얼 문서를 직접 검색하거나 타 부서에 문의하여 답변 준비
        - 수집한 정보를 바탕으로 이메일 초안 작성 후 전송
        
        > 각 요청사항에 대해 병렬적으로 처리하고, 각종 문서를 뒤져보거나 타 부서에 문의하는 과정을 거치면서 막대한 인지적, 시간적 자원이 소모되어 불편함을 겪음
        > 
    - 개선 후 시나리오
        - 고객사의 복합 문의 이메일 수령
        - 이메일 본문을 AI Agent의 입력값으로 제공
        - Agent가 의도 분류 → ERP 검증 및 수정 → RAG 검색 자동 병렬 처리
        - ERP 시스템 수정 관련 확인 요청 Slack 알림 수신 → 내용 확인 후 클릭 한 번으로 ERP 업데이트 승인
        - Agent가 생성한 최종 답변 텍스트를 검토한 후 고객에게 발송
        
        > 기존에 며칠이 걸리던 고객 문의 응대가 수 분 이내로 가능해지며, 담당자의 작업은 승인 + 검토 2단계로 축소
        > 
2. 고객사 담당자 시나리오
    - 페르소나: 고객사의 구매 담당자로서 각종 수정사항과 질의사항을 SDS에 문의하고 답변을 기다림, SDS의 응답 속도가 후속 업무 일정에 직접적인 영향을 미쳐 늦은 답변으로 인해 발주 변경의 실효성이 사라진 경험 보유
    - 개선 전 시나리오
        - SDS에게 문의 이메일 발송
        - 영업일 기준 며칠 동안 관련 처리 현황 불투명 상태로 대기
        - 답변 이메일 수령 후 관련 작업 처리 시작
        
        > 답변 수신 시 이미 출하가 완료되었거나 변경 처리 기한을 초과한 경우 발생해 불편함을 겪음
        > 
    - 개선 후 시나리오
        - SDS에게 문의 이메일 발송
        - 당일(또는 수 시간) 내에 처리 결과 및 명확한 정책 안내 답변 수령
        
        > 
        > 
        > 
        > 적시 처리를 통해 출하 전 변경 사항 적용 완료, 이후 물류 일정 차질 없이 진행
        > 

### 성공 지표

1. 요청 처리 평균 소요 시간: 기존 n일 → n분 이내로 단축
2. 요청 처리 정확도: 숙련된 담당자의 판단 하에 정확도 95% 이상 확보
3. 모델 지표: 라우터 정확도 99% 확보, RAG Context Recall 90% 이상, Text-to-SQL 쿼리 성공률 95% 이상

### 핵심 기능 목록

- **F-01. 복합 의도 라우팅:** 사용자의 입력을 분석해 ERP 제어(`ACTION_ONLY`), 단순 질의(`QA_ONLY`), 복합 요청(`BOTH`)으로 분류
- **F-02. 안전한 ERP 트랜잭션 자동화**(`ACTION_ONLY`)**:** 주문번호, 수량 등의 필수 파라미터를 추출하고, 가용 재고 및 납품 상태를 검증한 뒤 SAP OData API를 통해 시스템 업데이트
- **F-03. 매뉴얼 통합 RAG 검색**(`QA_ONLY`)**:** 다양한 사내 규정과 매뉴얼을 통합된 단일 Vector DB에서 탐색하여 지식 기반 답변 생성
- **F-04. Human-in-the-loop:** DB 업데이트 등 중요한 액션을 수행하기 직전, 담당자의 최종 승인을 거치는 안전망 제공
- **F-05. 최종 답변 텍스트 자동 생성:** F-02, F-03의 결과를 종합하여 비즈니스 이메일 형식의 답변 초안 자동 생성
- F-06. PDF/이미지 입력 처리: PDF/이미지 스캔본 OCR에서 텍스트를 추출하여 동일 파이프라인 처리하는 방향으로 확장 가능

# **🛠️ TRD**

### 개요

본 시스템은 LangGraph 기반 오케스트레이션을 중심으로, 정형 데이터(ERP DB)와 비정형 데이터(Vector DB)에 각각 특화된 두 개의 Worker 에이전트가 병렬 동작하는 하이브리드 구조

- **단일 LLM 환각 원천 차단:** 단일 LLM에게 모든 일의 처리를 맡길 경우, 권한 없는 DB를 조작하려 하거나 없는 재고를 상상해서 답하는 환각 현상 발생 가능성이 높으므로 도메인 분리 라우팅을 통해 정형 데이터(ERP DB)와 비정형 문서(RAG)의 접근 권한을 각기 다른 에이전트로 분리하여 안전한 응답 생성하도록 구성
- **규칙 기반 RPA 경직성 탈피:** 기존 규칙 기반 RPA나 단순 AI 분류기는 복합 의도를 종합적으로 추론하지 못해 실제 업무 처리 환경과의 괴리가 존재했으므로 조건부 라우팅을 통한 병렬 처리로 수량 변경과 같은 액션 수행과 위약금 확인과 같은 지식 검색을 동시에 수행하여 현업의 복잡한 커뮤니케이션 환경에 적합한 에이전트로서 기능하도록 구성

### **시스템 아키텍처**

- **오케스트레이션:** `LangGraph`
    - 복잡한 워크플로우 제어를 위해 상황에 따른 분기(Routing)와 상태(State) 기억이 필수적이기 때문에 LangChain이 아닌 LangGraph로 선택
- **LLM Model:**
    - *의도 분류 라우팅 & Worker A 정보 추출/Worker B 검색 키워드 추출:* `Qwen 2.5 (7B)` 또는 `Llama 3 (8B)` (무료)
    - *Text-to-SQL*: `Qwen 2.5 Coder` 또는 `gpt-4o-mini` 또는 `Claude 3.5 Haiku`
    - *최종 답변 합성:* `GPT-4o` 또는 `Claude 3.5 Sonnet` (유료)
    
    > 추가적인 fine-tuning 없이 few-shot 프롬프팅으로 진행하고자 함
    > 
- **RAG 스택:**
    - *Vector DB:* `ChromaDB`
    - *Embedding:* `text-embedding-3-small` (또는 무료 `multilingual-e5-small`)
    - *Reranker:* `bge-reranker`
- **Guardrail:**
    - `Pydantic` 유효성 검사
- **백엔드 및 시스템 연동 (FastAPI):**
    - 외부 증명용: SAP Business Accelerator Hub 샌드박스 API로 OData 통신
        - 샌드박스 API를 활용해 실제 ERP 필드를 수정하진 않지만 OData를 쐈을 때 성공적으로 처리됐음을 알리는 SAP의 응답을 받을 수 있어 테스트용으로 사용 가능
    - 내부 동기화용: 성공 응답 확인 후 로컬 SQLite DB 직접 업데이트

### 평가

- **라우터 정확도:** Input의 의도를 요청/질의로 정확히 분류하는 비율
    - classification 모델 평가지표 사용
- **Text-to-SQL 성공률:** 스키마를 바탕으로 올바른 쿼리를 생성하고 쿼리를 실패 없이 호출하는 비율
    - 이때 시간이 너무 오래 걸리는 쿼리도 실패로 처리
- **RAG 검색 품질 (Hit Rate / NDCG / Recall / Precision):** 답변 관련 문서 도출 정확도 비교
- **최종 생성 텍스트 품질:** 별도의 평가 LLM(LLM-as-a-Judge)으로부터 Faithfulness(hallucination X), 답변 정확도 평가
- **End-to-End Latency:** 이메일 입력부터 DB 업데이트 및 최종 답변 텍스트 생성까지 걸리는 총 소요 시간

### 예외 처리 및 방어 로직

- **Guardrails 예외:** 추출된 수량이 음수이거나 필수 값(주문번호 등)이 누락된 경우, Worker A에서 자체적으로 LLM에게 재추출을 지시하거나 사용자에게 명확한 정보 입력을 재요청
- **요청 처리 불가 상태:** 테이블 조회 결과 가용 재고가 부족하거나 이미 완료 처리된 주문인 경우 등 요청을 처리할 수 없는 상태임이 확인된 경우, API 호출을 중단하고 `요청 처리 불가`라는 상태 코드를 넘김
- **Human-in-the-loop:** 실제 DB 업데이트(FastAPI 호출) 직전, LangGraph의 `interrupt` 기능을 활용해 담당자에게 Slack 승인 알림을 발송하고, 사람의 승인이 떨어져야만 트랜잭션이 실행되도록 통제
